### 一、Cordova 应用热更新的原理
![](https://cdn.nlark.com/yuque/0/2023/png/25743026/1698390115860-aec0b118-fa52-405c-835b-c9b563e1fdda.png)	

上图是 Cordova 应用的基础架构，在这个架构中，Cordova Plugins 是指基于 Cordova 插件模式开发的一系列原生插件，这些插件提供了与 OS 交互的能力，在这个基础上，Cordova 提供一个 WebView，这个 WebView 用于加载 WebApp 的内容，并提供与原生插件通讯的能力，开发者可以通过 js bridge 访问原生插件，并通过插件与系统交互。这算是在 webview 的沙箱模式中打开操作系统的个口子。

基于 Cordova 这种模式，一般有两种使用场景：

+ 将 Cordova 作为一个扩展插件，在原生 APP 中加入 Cordova 作为扩展插件，部分不需要原生开发的功能使用 Cordova+前端技术的方式来开发。  

+ 将 Cordova 作为原生的基础能力，所有业务功能全是使用前端技术来开发，用 WebView 来承载用户的操作界面，所有功能全部由前端完成。这种做法不需要考虑多端开发的问题，社区有大量的安卓和 ios 插件可供使用，在业务初期或者简单的场景通过这种方式可以大大的减少开发成本。这也是目前我们的使用方式。



### 二、为什么需要热更新
主要有三点：

+ 发布的便利性，不需要发布软件版本，甚至不需要用户同意，就可以直接更新。在软件研发过程中，使用瀑布流之类的研发模型，虽然更新迭代缓慢，软件的发布可以不通过热更新，但是仍需实时修复发现的小 bug 的能力。而敏捷研发模型中不仅需要快速修复 bug 的能力，同时也需要一种简单快捷的发布方式，1~2 周 1 个迭代的速度，让应用商店的审核时间显示无比漫长
+ 方便版本管理和团队协同，在一个项目中，特别是大型的项目(支付宝 APP、淘宝 APP 之类)，同时会有多个团队协同开发，分管不同的功能模块，迭代进度也不相同，每个团队都自行发版的话，版本管理势必混乱，如果是一起发布的话，那协同的成本就太大了。
+ 用户体验，如果每周更新一次或多次，每次更新都需要到应用市场手动下载更新，用户肯定要疯了。



### 三、热更新原理
基于 Cordova 这种模式，用户界面、业务逻辑等等都是有前端来实现的，前端开发完成后将代码打入到 APP 中，然后 APP 在启动时会默认启动 webView 并加载入口文件(这个入口文件是可以配置的，一般都是配置 index.html)，其实就像我们在 PC 本地打开自己编写的 html 页面一样。

那么如果我们要更新前端，不需要像原生应用那么麻烦，只要替换掉前端代码就行了，以我们内部的实现为例，App 在打包时，会将前端代码打包成一个压缩包，App 打包成功后，生成 apk/ipa 包，安装到手机上后，会生成一个 app 的安装目录，前端相关的代码会解压到 www 目录中，app 运行的时候，WebView 会加载`www/index.html`。



### 四、热更新实现
简单来说我们的热更新就是做一件事：把目前 app 的安装目录中的 www 目录下的前端代码替换掉。分解出来就是两个步骤：

+ 服务端下发代码
+ app 端接收并替换文件



#### 1.服务端如何识别APP身份
APP 发送请求，但是服务端怎么识别 APP 身份，获取正确的 APP 代码？ 如果线上有多个 APP，而热更新的服务都是同一个（大部分都是这方式），那么需要通过一些标识来识别 APP。一般我们会为每个 APP 分配一个唯一的 Code，称为AppCode，这个 Code 会根据 APP 名称或作用或特性来命名。



#### 2.<font style="color:rgb(51, 51, 51);">APP端怎么判断是否需要更新呢？</font>
这里就需要引入版本管理。特别是存在多个有差异的生产包的情况，如果没有版本管理，代码的更新和迭代会陷入混乱。

一般来说我们会把版本划分成两种： **大版本**，在 APP 打包的的时候，我们会 APP 设置一个三段式的版本号(x.x.x)，这个版本会被写入在原生层，可以通过原生层的接口获取，这个版本除非重新打包否则是不会变化的，这个固定的版本，我们称为大版本。一般只有在原生层有变更（添加/减少/更新插件）或者前端代码有重大调整（代码大面积重构之类的）的时候才会更新大版本。

**小版本**，在大版本之上，我们加入另一个版本版本，这个版本用来表示热更新的版本，我们称之为web 版本或小版本，一般小版本用一个递增的数字来表示，比如第一个版本是 01，下一个版本就是 02，小版本依托大版本，一个大版本会有多个小版本，在正常情况下，大版本只会有一个活跃的小版本。小版本的发布就比大版本活跃多了，基本上 bug 修复、迭代发布都是发布小版本。

这样，一个完整的版本表示如下：x.x.xxx 例如：

+ 1.0.000 表示大版本为 1.0.0，小版本为 00 则表示没有热更新过
+ 2.0.002 表示大版本为 2.0.0，小版本为 02



#### 3.更新包
热更新时下发的代码包，我们一般称为更新包，更新包一般都是一个代码的压缩包。热更新时服务端下发压缩包，APP 端获取并解压到 www 目录。这里，我们需要把更新包存放到服务端，但是这会对服务端造成一定的性能影响，而且如果容器被重置或者回收，存放的文件可能也会丢失。所以一般我们会选择一个文件存储服务来存放更新包，目前内部用的是阿里 OSS。

那么我们的热更新操作可以变成：

发布：

+ 本地构建完代码
+ 压缩构建完的代码
+ 上传到 OSS



更新：

+ 服务端下发代码
+ app 端接收并替换文件



#### 4.服务端设计
服务端的主要作用是下发热更新版本和相关的更新包，首先我们需要一个保存热更新记录的地方，这里当然是用数据库啦，由于更新包是放在文件存储服务中，所以只要保存包的 url 就行。  


##### 热更新表设计
```sql
CREATE TABLE `app_web_version` (
  `id` char(36) NOT NULL COMMENT 'id',
  `app_code` varchar(50) NOT NULL DEFAULT '' COMMENT '应用code',
  `web_version` varchar(20) NOT NULL COMMENT 'web版本号（小版本）',
  `app_version` varchar(20) NOT NULL COMMENT 'app版本号（大版本）',
  `app_version_code_ios` varchar(20) NOT NULL COMMENT 'app版本_iOS',
  `app_version_code_android` varchar(20) NOT NULL COMMENT 'app版本_android',
  `download_url` varchar(500) NOT NULL COMMENT '更新包下载url',
  `update_info` varchar(500) DEFAULT NULL COMMENT '更新说明',
  `release_date` datetime NOT NULL COMMENT '发布日期',
  `created_on` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录创建时间',
  `created_by` char(36) DEFAULT NULL COMMENT '记录创建者Id',
  `modified_on` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '记录修改时间',
  `modified_by` char(36) DEFAULT NULL COMMENT '记录修改者Id',
  `is_deleted` tinyint(1) DEFAULT '0' COMMENT '是否已删除',
  PRIMARY KEY (`id`),
  KEY `idx_app_code` (`app_code`),
  KEY `idx_app_code_app_version` (`app_code`, `app_version`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8 COMMENT='APP web 版本表';
```

##### 更新说明
这个表用来存放热更新的发布记录，每次发布都会产生一条记录。我们来看下表的设计： `app_code`、`app_version`分别用于识别 APP 和对应的大版本，这两个字段与`web_version`要组合成一个唯一的记录：每个 APP 的每个大版本下的小版本都是不重复的（所以小版本是递增的）。 is_deleted表示该记录是否已删除，这里是一个逻辑删除的设计，如果小版本被删除，记录还是会保留下来的，这样会方便后续查看发布记录和回溯版本。

##### 发布时间
在其他 APP 上，不管是热更新还是大版本更新都会有一些版本更新说明 `update_info`，这个说明也是需要跟随版本下发的。

虽然我们有一个 `created_on` 来表示记录创建时间，但是这里会有一个问题：这个是记录添加到数据库的时间，基本可以等于版本发布的时间，但是如果多团队协作和代码迭代合并速度比较快的话，在版本发布时，可能代码已经又更新过了，如果我们要找到这个版本真实发布时间，从这个时间反推代码提交的节点或者记录一个真实的发布时间 `release_date`。

##### 设备类型
表结构中我们用一个 `app_version` 字段来表示大版本。当然这里的设计还是有缺陷的，众所周知移动端系统是安卓和 IOS（其他的直接无视），而前面我们也说了，大版本一般是原生层有调整才发布的，而由于安卓和 IOS 在系统层面的不一致，有一些问题是一个系统有一个系统没有，可能需要对单独系统发布新的版本，如果原生层的问题会导致重大问题，我们会强制用户升级版本。而升级后的 APP 虽然是新的版本，但是前端的代码与旧版本是一致的，这时候如果需要发布热更新就得发两个版本，但是对于热更新来说，两个版本的代码又是一致的。所以我们最好把平台和版本做个解耦。这里，我们把版本拆分成两个与平台相关的字段： `app_version_code_ios` 和 `app_version_code_android` 。这样就可以根据平台管理版本了。

#### 5.添加版本记录
基于上面的版本表，我们发布一个版本，就是添加一条记录，为了保证每个大版本只有一个活跃的小版本，还需要删除之前该大版本下的小版本，而且还要让小版本号自增，这块我们可以用 sql 来实现：

```sql
-- 1、 获取新的热更新版本号 --
select @curVersionCode := ifnull(max(version_code), '00')
from app_web_version
where app_code = 'AppCode'
and is_deleted = 0
and (app_version_code_ios = 'AppVersion' or app_version_code_android = 'AppVersion')
;

select @newVersionCode := case when @curVersionCode + 1 > 9 then @curVersionCode + 1 else lpad(@curVersionCode + 1, 2, '0') end
;


-- 2、 作废当前大版本之前的热更新版本 --
update m_app_web_version
set is_deleted = 1, modified_on = now(), modified_by = 'system'
where app_code = 'AppCode'
and is_deleted = 0
and (app_version_code_ios = 'AppVersion' or app_version_code_android = 'AppVersion')
;


-- 3、 生成当前版本的新热更新版本 --
insert into m_app_web_version (id, app_code, web_version, app_version_code_ios, app_version_code_android, download_url, update_info, update_info_url, release_date, created_on, created_by, modified_on, modified_by, is_deleted)
values (uuid(), 'AppCode', @newVersionCode, 'AppVersion', 'AppVersion', 'url', 'updateInfo', 'updateInfoUrl', 'releaseDate', now(), 'system', now(), 'system', 0)
;
```



#### 6.数据下发
在整个热更新的过程中，服务端只负责根据 APP 端提交的参数获取对应的数据版本，并处理更新说明中的换行符以便前端可以简单的做到显示处理。

+ 版本获取： 服务端只要提供一个get-web-version的接口即可，接口要求传入三个参数：AppCode、version、platform，这里需要根据platform生成版本的查询字段app_version_code_${platform}，查询的 sql 如下：

```sql
select id, web_version, download_url, update_info,update_info_url from app_web_version where app_code='AppCode' and pp_version_code_{platform} = 'version' and is_deleted = 0;
```

前面提到过，一般一个大版本只会有一个活跃的小版本，所以我们加上is_deleted=0来获取活跃的版本。 这里还要做个处理，如果获取不到记录，直接返回空就行了。



+ 更新说明处理：更新说明一般都是多行的，格式如下:
    1. xxxxxx
    2. xxxxxxx
    3. xxxxxxx  
换行符虽然可以存入数据库，下发到 APP 的话，需要把更新说明处理成单行才行，但是 APP 在展示的时候需要恢复成换行的，那么我们需要把换行符替换成特殊字符，这里我用的是|来替换：



接口下发的数据如下：

```javascript
{
  "webVersion": "01",
  "downloadUrl": "xxxxxx",
  "updateInfo": "1.xxxxxx|2.xxxxxxxx|3.杀了主程祭天",
  "updateInfoUrl": "xxxxxxxx"
}
```



#### 7.APP端实现
APP 端需要完成以下操作：

1. 获取本地大版本信息和 AppCode
2. 通过接口获取最新的热更新版本
3. 判断是否需要更新版本
4. 如果需要更新，下载热更新包
5. 解压更新包并替换到 www 目录
6. 加载新代码



##### 热更新执行的时机
由于我们的接口会直接返回热更新的代码包，为了不让外界能够随意的获取到我们的代码，接口需要下发代码包之前会验证请求的用户身份，所以一般热更新操作都是在用户登录成功后执行。这个是用户手动登录后执行，用户登录成功后一般都会保存登录态，在用户下次打开 APP 时不需要重新登陆，所以我们还要在 APP 初始化的时候加登录态判断，如果用户已登录，也要执行热更新操作。还有一个场景，用户已经打开 APP 并且已登录，这时我们发布一个热更新，用户要到下次打开 APP 或者重新登录才会更新，这里的话，我们添加更多的时机判断，比如在切换 tab 的时候每小时（时间间隔看具体的需求）判断一次是否需要更新、在 APP 从后台切到前台的时候做判断。



##### AppCode 存储和获取
虽然大版本是跟随原生层的，但是AppCode怎么获取呢？一般来说每个 APP 的 code 都是固定的，这个可以在前端代码构建时写入，也可以将写在一个不变的文件中来实现（这个方法有很多，就不概述了）



##### 小版本的存储和获取 
在判断本地代码是否需要更新的时候，需要将在接口返回的webVersion和本地的webVersion来做比较，如果本地版本小于接口返回的那就执行更新操作。因为webVersion跟AppCode不一样，是要实时变化，不适合直接存储在文件中（虽然我们能用原生能力读写文件，但是这个操作还是太麻烦了）。在这里我们直接用localstorage来存储就行了，存储的数据格式如下：

```javascript
{
    appVersion: '3.0.0',
    webVersion: '02',
    fullVersion: '3.0.002'
}
// 存储
localstorage.setItem(
    'version',
    JSON.stringify({
        appVersion: '3.0.0',
        webVersion: '02',
        fullVersion: '3.0.002'
    })
);
```



##### <font style="color:rgb(51, 51, 51);">小版本比较</font>
我们的小版本都是用字符串来存储和表示，那么直接比较可能会出现'11'大于 '102'的情况，所以我们规定所有的小版本必须由数字组成，不能带其他类型的字符，在比较时转成整数来比较。



##### 更新包下载
下载这块我们直接使用原生的下载插件来完成下载和保存即可，因为这个下载和保存的操作是静默的，更新过程中用户只会看到一个更新的进度条。通过前端的方式来完成文件下载和写入，会被浏览器的安全策略限制，导致体验很差（不管怎么实现，都需要用户确认才能下载，这个在 APP 中就显的很诡异了）。



##### 更新代码
解压的操作，node 可以实现，不过在浏览器端就算了，安全策略不会让你直接访问本地文件，所以还是只能借助原生插件的能力。在解压的时候，我们直接把代码解压到 www 目录，这样可以直接覆盖替换文件。



##### 加载新代码
前端的代码是运行在浏览器中的，加载index.html后，会把相关的代码文件载入到内存中（简单的理解就是有一个虚拟的运行目录），在代码更新后，这时候在浏览器运行的还是旧的index.html和代码文件（载入内存后，即使原始文件修改也不会影响）。我们要加载新的代码，最简单的方法就是刷新当前页面，不过这里有需要避讳的地方，我们先看代码:

```javascript
function reOpenIndexPage() {
  return new Promise(resolve => {
    window.location.href =
      window.location.href.replace(/#.*$/, '') + '?v=' + new Date().getTime()
    // 阻止程序继续往下执行，以免出现问题
    window.setTimeout(() => {
      resolve()
    }, 1000 * 60 * 60)
  })
}
```

	

这方法返回一个Promise，热更新是在登录后才执行，在 APP 中用户登录后是有一系列的操作的，如果有热更新的话，最好是不让这些操作执行，因为在更新成功后会重载页面，当前的操作都会被取消，这些操作会变成无效的消耗，如果这时候正在下载数据或者做一些关键的数据提交，会因为重载被中断，那就有可能导致问题。结合上面的思考和热更新本身就是异步操作，那就让所有其他操作都在热更新之后执行，如果没有热更新，则继续执行，如果有热更新，就更新并重载页面。在重载时，为了防止后续的操作继续执行，这里加了个定时器来阻塞后续操作（如果直接返回错误的话，会触发一些异常处理的操作和上报错误的操作，这也不是我们想要的）。

<font style="color:rgb(51, 51, 51);"></font>

<font style="color:rgb(51, 51, 51);">细心的同学会发现我们的刷新操作是这样的：</font>

```javascript
window.location.href =
  window.location.href.replace(/#.*$/, '') + '?v=' + new Date().getTime()
```

因为在 APP 的方式，一般都是通过本地的 file 协议来访问前端资源的，这里安卓和 IOS 是有区别的，IOS 有wkwebview的限制只能通过本地服务器的方式来访问，也就是走http://localhost:port的方式，但是安卓只能走 file 协议，使用 file 协议是无法使用路由的 history 模式，只能使用 hash 模式。上面代码的正则就是去掉#和后面的路径，这里是为了让页面回到首页刷新，如果在当前页面刷新，当前页面不是顶级页面，在当前页面刷新后，返回按钮可能会无法返回到上一个页面（history 被清掉），当然还有一个更严重的问题，如果当前页面依赖上一个页面在加载时获取的数据，那就直接报错了。后面的v参数是为了防止缓存问题。



#### 8.版本发布
前面我们说了整个热更新流程的前后端实现，那么最重要的一点就来了，我们怎么发布代码？？虽然我们前面给出了热更新发版本的 sql，但是发布热更新版本还有很多操作要做，以现有的 Vue 的项目为例：

1. 构建代码
2. 压缩 dist 目录，生成压缩包
3. 上传压缩包到 oss
4. 编辑更新 sql
5. 执行 sql 发布版本

看起来很简单，但是所有操作都是手动操作，如果一个环节出错，就会引起线上事故。最好的方式还是把这个流程自动化。这里有两种方式，一种是在线的，一种是本地的在线的可以借助 Jenkins 这类工具或者自己部署个的 node 环境来完成 git 代码拉取和打包构建，本地的方式就是在通过脚本在本地完成这些操作，其实都是操作都是差不多，都是需要脚本来完成生成压缩包，上传到 OSS 的操作。在发布版本的时候有些区别，本地的方式可以直接生成 sql 文件，开发人员拿着 sql 去发布就行了，在线的就不能直接输出 sql 了，最好的做法让服务端提供一个发布接口，构建打包完成，把数据提交到服务端的接口，并告诉服务端更新的 AppCode 和大版本，由接口来完成数据更新。



##### 打包格式抉择
目前我们使用的是7z的压缩格式，之前使用的是zip的，经过我们的对比同样的文件 7z 压缩比 zip 压缩后的体积要小 40%以上，而解压效率没受到什么影响。

来看看 7z 打包的实现：

```javascript
/**
 * 生成临时目录
 */
function createTempDist() {
  fileName = `hot-update-${formatTime(+new Date())}`
  const hotUpdataPath = path.resolve(appPath, fileName)

  // 生成临时目录
  shelljs.mkdir('-p', hotUpdataPath)
  shelljs.cp('-R', path.resolve(appPath, 'dist/*'), hotUpdataPath)

  return hotUpdataPath
}

function make7z() {
  const distPath = createTempDist()
  const _dist = distPath + '.7z'

  return new Promise((resolve, reject) => {
    var myStream = node7z.add(_dist, distPath, {
      recursive: true,
      $progress: true,
      $bin: pathTo7zip,
    })

    myStream.on('data', function(data) {
      console.log(data) // ? { status: 'extracted', file: 'extracted/file.txt" }
    })

    myStream.on('progress', function(progress) {
      console.log(progress) // ? { percent: 67, fileCount: 5, file: undefinded }
    })

    myStream.on('end', function() {
      // end of the operation, get the number of folders involved in the operation
      myStream.info.get('Folders') // ? '4'
      console.log('=========>>>>> 成功生成7z压缩包')
      // 删除临时目录
      shelljs.rm('-rf', distPath)
      resolve({ distPath: _dist, fileName: fileName })
    })

    myStream.on('error', err => {
      console.error(err)
      reject()
    })
  })
}
```



这里主要是使用了node-7z来完成 7z 格式的打包。

function doUpload(path, key) {
  return new Promise((resolve, reject) => {
    return fs.readFile(path, function(err, code) {
      if (err) {
        console.log(err)
        return Promise.reject(err)
      }

      var url = `http://oss-cn-shenzhen.aliyuncs.com/xxxxx/${key}`

      var params = {
        Bucket: 'xxxxx',
        Key: key, // 注意, Key 的值不能以 / 开头, 否则会返回错误.
        Body: code,
        AccessControlAllowOrigin: '*',
        ContentType: 'application/octet-stream',
        CacheControl: 'no-cache', // 参考: http://www.w3.org/Protocols/rfc2616/rfc2616-sec14.html#sec14.9
        ContentDisposition: '', // 参考: http://www.w3.org/Protocols/rfc2616/rfc2616-sec19.html#sec19.5.1
        ContentEncoding: 'utf-8', // 参考: http://www.w3.org/Protocols/rfc2616/rfc2616-sec14.html#sec14.11
        ServerSideEncryption: 'AES256',
        Expires: null, // 参考: http://www.w3.org/Protocols/rfc2616/rfc2616-sec14.html#sec14.21
      }

      return oss.putObject(params, err => {
        return err ? reject(err) : resolve(url)
      })

  })

##### 发布管控
前面聊到的服务端发布或者是 sql 提交的发布，都有一个缺陷，就是只要执行了就能发布，也就是只要有构建的权限在什么时间都能直接发布，这个是十分危险的，如果发布的代码不正确或者在上班时间发布可能会引起重大事故。因此，在发布流程上我们需要引入审核机制，提交发布后必须由更高权限的人审核通过才能完成真正的发布。



#### 9.增量更新
从前面描述的热更新的方案，我们知道实际上就是代码替换，那么是不是每次只要替换有变化的文件和 index.html 就行了？

这个是有办法实现的，git 有一个 diff 命令，可以通过与其他 commit 对比获取有变更的文件，那么我们只要保存一个 commit 的 hash，每次都用这个 hash 来做比较，一段时间更新下 hash，就可以一直获取指定 hash 之后有变更新的文件，打包的时候只打包这些文件，这样就实现能实现增量更新，并且大幅度减少更新包的体积。

当然，理想很丰满，现实很骨感，在以前 jq 或者其他建议框架的项目可以实现，因为文件名基本都不会变，虽然会打包和压缩，也可以做到不改变文件和资源的名称。但是在 webpack 搭建的脚手架上面，我们最终是要打包 dist 目录，一般 dist 目录都是不提交的，就算提交了，dist 目录里面，代码文件和资源基本都是重命名过的，diff 比较不出来，特别是代码的分片文件，每次都是新的 hash 名称。如果要实现增量更新，那就得做分片命名，或者让打包的代码的名字保持不变，方便 diff 比较。这块实现的成本比较高，收益不确认所以内部暂时还没实现 。



### 五.扩展方式
#### 1.灰度发布
测试的同学在测试 APP 的时候经常会遇到一个问题，代码在测试环境测试通过，不过生产环境怎么样不清楚，最好的方式就是去生产测试，而生产环境一发布就会覆盖所有用户，就算能测出问题也无法终止发布，在这种情况下我们只能把发布时间定的很晚，发布后测试同学连夜测试，有问题再回撤，如果是重大更新，估计大家都很难下班。

这里我们引入灰度发布的方式，主要就是在热更新的基础上加上匹配的规则：

```sql
CREATE TABLE `app_web_version` (
  `id` char(36) NOT NULL COMMENT 'id',
  `rule` text COMMENT '灰度规则',
  `app_code` varchar(50) NOT NULL DEFAULT '' COMMENT '应用code',
  `web_version` varchar(20) NOT NULL COMMENT 'web版本号（小版本）',
  `app_version_code_ios` varchar(20) NOT NULL COMMENT 'app版本_iOS',
  `app_version_code_android` varchar(20) NOT NULL COMMENT 'app版本_android',
  `download_url` varchar(500) NOT NULL COMMENT '更新包下载url',
  `update_info` varchar(500) DEFAULT NULL COMMENT '更新说明',
  `update_info_url` varchar(500) DEFAULT NULL COMMENT '更新说明URL',
  `release_date` datetime NOT NULL COMMENT '发布日期',
  `created_on` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录创建时间',
  `created_by` char(36) DEFAULT NULL COMMENT '记录创建者Id',
  `modified_on` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '记录修改时间',
  `modified_by` char(36) DEFAULT NULL COMMENT '记录修改者Id',
  `is_deleted` tinyint(1) DEFAULT '0' COMMENT '是否已删除',
  PRIMARY KEY (`id`),
  KEY `idx_app_code` (`app_code`),
  KEY `idx_app_code_app_version` (`app_code`, `app_version`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8 COMMENT='APP web 版本表';
```

增加一个rule字段来存放规则，这个规则要求是一段 JSON 字符串，比如想要为指定用户发布热更新只需要传入规则：

```json
{
  "account": ["xxxx"] // 数组可以支持多个账号的匹配
}
```

这样服务端就可以根据请求的数据判断是否命中规则，命中则下发热更新版，没命中则不下发。

#### 2.灰度更新模式2
上面说的灰度模式有一个致命的缺陷，因为每个热更新版本只会有一个活跃的小版本，那么发布了灰度版本之后，线上就只有一个灰度版本，如果这时候有一个代码不是上一个热更新版本的用户或者是新安装的用户打开 APP，那么该用户就无法更新到最新的代码，会导致用户无法使用或者是大量的数据问题等等。

这个模式虽然解决了测试同学的问题，但是又引出新的问题，我们得升级这个灰度模式。要解决上面的问题，我们需要让一个大版本同时存在多个活跃的小版本，但是这么一来，服务端就无法知道要下发哪一个版本，所以还需要对小版本做优先级划分，根据优先级来返回命中的版本。

先加多一个优先级字段：

```sql
CREATE TABLE `m_app_gray_web_version` (
  `id` char(36) NOT NULL COMMENT 'id',
  `rule` text DEFAULT NULL COMMENT '规则',
  `app_code` varchar(50) NOT NULL DEFAULT '' COMMENT '应用code',
  `web_version` varchar(20) NOT NULL COMMENT 'web版本号（小版本）',
  `app_version` varchar(20) NOT NULL COMMENT 'app版本号（版本）',
  `download_url` varchar(500) NOT NULL COMMENT '下载url',
  `update_info` varchar(500) DEFAULT NULL COMMENT '更新信息',
  `update_info_url` varchar(500) DEFAULT NULL COMMENT '更新信息 URL',
  `sort` int(11) DEFAULT NULL COMMENT '规则判断顺序',
  `created_on` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录创建时间',
  `created_by` char(36) DEFAULT NULL COMMENT '记录创建者Id',
  `modified_on` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '记录修改时间',
  `modified_by` char(36) DEFAULT NULL COMMENT '记录修改者Id',
  `is_deleted` tinyint(1) DEFAULT '0' COMMENT '是否已删除',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8 COMMENT='APP web 版本表';
```



新增一个sort字段来表示优先级，获取的时候会根据sort做升序排序，然后服务端一次匹配灰度规则，如果灰度规则为空则表示是一个通用的版本。简单描述下这种灰度模式的流程：
