#### 一、背景
##### 1.离线应用的基本原理
1. 在用户有网络的时候，把可能用到的数据全部下载到客户端。
2. 客户端使用过程中访问的都是本地数据，摆脱对网络的依赖。
3. 客户端再次连接网络的时候，把产生的本地数据全部上传到服务端。



##### 2.具体实现
在客户端和服务端各有一个数据库，客户端访问和操作的都是本地数据库 即 SQLite。

重点问题在：客户端和服务端的数据要及时流通。

做这件事的就是我们今天要说的：同步数据服务。

![画板](https://cdn.nlark.com/yuque/0/2023/jpeg/25743026/1698913925156-26358412-c9d4-4aa7-908c-e35d99134882.jpeg)



##### 3.复杂性
看似简单的事情，事实上并不简单。绝大多数导致复杂的原因都是：**<font style="color:#DF2A3F;">数据量大！</font>**

眼睛亮的人可能已经看出来了，前面说的可能用到的数据是怎么判定的？

事实是：根本判断不了。因为无法预测用户的行为，所以项目初期的做法就是**<font style="color:#DF2A3F;">下载所有的数据</font>**。



##### 4.手段
很快问题就出现了，数据量太大了。不仅仅是用户下载的时间变长了，服务器的查询也吃不消。

所以有了数据分割，初步我们把数据分为：基础数据、业务数据。

后来随着项目迭代，又加入了项目、模块、楼栋的划分。

而且还进行了分页，数据量多了之后要分页下载。



##### 5.过程
第二个问题，这么大量的数据，用户每次同步的时候都要下载吗？

这显然不现实，所以需要增量下载，因此需要记住每种数据的同步记录。

每种数据的分类就是上面说的数据划分，比如：

A项目，基础数据

B项目，基础数据

A项目，M模块的业务数据

增量下载基本的原理是：**<font style="color:#DF2A3F;">客户端记住各类数据的下载时间，再次下载同类数据的时候，只向服务端请求这个时间点之后的数据。</font>**

根据各类数据的划分，添加了对应的时间点。



##### 6.问题
+ 能不能处理
    - 数据冲突
+ 处理哪些数据：
    - 基础数据
    - 业务数据
    - 待办数据
+ 处理多少数据
    - 项目
    - 模块
    - 楼栋
+ 处理时机
    - 自动触发
    - 手动触发
+ 处理方式
    - 分批(队列)
    - 增量(时间点)



##### 7.数据划分
按照业务场景，数据可分为

+ 基础数据：操作过程中一定会用到的数据。可再次细分为
    - 无项目基础数据
    - 项目基础数据
+ 业务数据：一般指用户行为产生的数据
+ 待办数据：严格来说是没有待办数据的，这里说的待办是指基础数据 + 业务数据，但下载多少基础数据和多少业务数据是由待办的视角去划分的。比如用户在待办下拉时，会下载3个项目的基础数据，100条业务数据，这些取决于客户的待办。



##### 8.处理方式
###### 队列
划分数据后，数据量依然很大。如果一次性返回数据，可能会非常慢。所以要做分页，离线的分页要用到队列的方式。

队列可以理解为：同步数据的任务栈



为什么需要队列

    - 数据量大，需要分页
    - 同步数据是一个过程，如果过程被中断，需要支持断点续传



下载数据的过程，就是不停执行队列的过程。



队列分为三种：

    - 上传队列
    - 下载队列
    - 图片队列



根据数据类型，下载队列又分为：

    - 下载基础数据队列
    - 下载业务数据队列



###### 增量(时间点)
通过记录时间点，实现数据的增量下载。即每次执行下载后，会记录一个时间点，下载再执行同样的数据同步时，只会同步上一次时间点之后的数据。

##### 4.上传数据
客户端进行数据操作之后，会记录需要上传的内容，等到有网络的时候，再一起上传。记录需要上传内容的叫做上传队列。

上传队列负责告诉程序哪个表的哪条数据需要上传，在执行上传的时候，就会去对应的表查询到对应的数据，提供给接口。

当然，事实不会这么简单，一个问题的数据大多不是存在一张表，所以上传队列需要分组。

上传队列的内容如下：

| id | relation_table_nane | relation_id | relation_column | time | group_id |
| :---: | :---: | :---: | :---: | :---: | :---: |
| 主键 | problem | xxx-xxx | id | xxx | xxx |
| 主键 | problem_xx | xxx-xxx | problem_id | xxx | xxx |




一条数据的查询语句大概是

```sql
select * from ${relation_table_name} where ${relation_column} = ${relation_id}
```



离线应用最复杂的场景大概就是上传了，需要处理数据版本，数据冲突，附件处理。

#### 三、优化方案
##### 1.现状
从一开始粗暴的全部下载，到数据切分和按需下载，确实缓解了很大的数据下载压力。但是问题的根本原因还在：数据量大。面对大量的数据的时候，一页一页地下载数据需要很多的时间。当划分到最小粒度之后，数据量依然很大的时候，问题还是出现了。

移动质检是以项目的维度录入数据，所以数据划分到项目级别之后，已经不能再细分了。



##### 2.痛点
线上有用户抱怨，登录之后的初始化时间太久(10~30分钟)，必须一直等待，无法进行操作。我们知道这是在下载基础数据，我们也知道基础数据无法再切分了。目前的方案已经看不到可以优化的地方了。



##### 3.分析
难点在于数据量太大导致的后端查询压力巨大，分页之后有导致请求过多，http 请求资源浪费。所以想出了数据包下载方案，借助于一个高速的前端导出方法，前端可以快速导入数据。

云创提供的一个导入数据方法，从 `*.json` 文件导入数据。后端提前把数据做成 json 文件，然后压缩成 zip 包，一次性下载后解压，然后导入。这样，即可以节省后端的数据查询时间，也节省了分页导致的 http 请求浪费的问题。

因为 json 文件并不能做到事实生成，所以有时效性的问题，暂时只考虑把基础数据（基础数据改动较小）做成数据包。

经过我们的测试，可以大大缩短下载数据的时间。



使用数据包下载数据的方式，可以缩短下载数据时间15 - 60倍。

举例xx租户的材料类型表的测试结果

**<font style="color:#DF2A3F;">使用队列下载的时间大概是: 1.5min - 5min。</font>**

**<font style="color:#DF2A3F;">使用数据包下载，只要不到7s。</font>**



用户经常返回初始化时间太长，有时候会达到10min或30min，是因为初始化的过程包括：

    - 下载无项目基础数据
    - 下载项目基础数据

**<font style="color:#DF2A3F;">使用数据包的方式下载，可以将15min 缩短到 1min左右。</font>**



##### 4.方案
对比队列的下载方式，数据包下载只有一次后端请求和一次文件下载，而且经过压缩后，消耗的流量也大大减少了。

数据包的下载方式，前提是：必须要有数据包，所以需要后端预先生成数据包。

生成数据包的策略可以多种，比方说：

+ 定时生成，时间间隔可以根据情况而定
+ 发布版本时生成，时间间隔不宜过久，不然会产生过多的增量数据。
