##### 1、Rewrite
在 Nginx 配置中，rewrite 是一个重写 URL 的指令。它可以将一个 URL 重写成另一个 URL，或者修改一个 URL 的查询参数、协议等内容。

下面是一个 rewrite 的示例：

```plain
rewrite ^/users/(.*)$ /profile?username=$1 last;
```

这个规则会将以 /users/ 开头的 URL 重写成 /profile?username=xxx 的形式，其中 xxx 是原 URL 中 /users/ 后面的部分。

除了重写 URL，rewrite 还可以修改请求的一些其他属性，例如协议、请求方法等。下面是一个修改请求方法的示例：

```plain
if ($request_method = GET) {
    rewrite ^/submit$ /submit.html break;
}
```

这个规则会将 GET 请求的 /submit URL 重写成 /submit.html，从而让 Nginx 返回 submit.html 文件。如果请求方法不是 GET，则不会执行重写操作。

需要注意的是，过度使用 rewrite 可能会影响 Nginx 的性能。因此，在编写 Nginx 配置文件时，应该尽量避免使用复杂的 rewrite 规则，以免影响网站性能。

六、如何配置反向代理

proxy_pass 是 Nginx 配置中用来反向代理的指令，它的作用是将客户端的请求转发到另外一台服务器上。具体来说，proxy_pass 指令将客户端发来的请求转发到后端服务器，并将后端服务器的响应返回给客户端。

```bash
location /api {
	proxy_pass http://localhost:3000;
}

```

当客户端发送请求 /api 时，Nginx 会将请求转发到 http://localhost:3000，并将后端服务器的响应返回给客户端。这个功能非常有用，因为它允许我们将客户端的请求分发到多个后端服务器上，以达到负载均衡的目的。

需要注意的是，proxy_pass 指令只能用于 location 块中，因为它需要根据不同的 URL 路径来进行不同的代理。另外，proxy_pass 还可以与其他的代理相关的指令一起使用，例如 proxy_set_header 用于设置请求头，proxy_cache 用于缓存请求结果等等。



七、常用配置

##### 1、try_files
用于在服务器上寻找文件并返回。

该指令的语法如下：

```plain
try_files file ... uri;
```

其中，file 表示要查找的文件，可以有多个，以空格分隔；uri 表示如果没有找到任何文件，则跳转到的 URI。

当客户端发送请求到服务器时，nginx 会按照指定的文件列表顺序依次查找文件。如果找到了文件，则直接返回给客户端；如果找不到，则继续查找下一个文件，直到找到为止。如果所有文件都查找完毕仍然没有找到，则会跳转到指定的 URI。

例如，以下配置将先尝试寻找静态文件 /usr/share/nginx/html/images/logo.png，如果找到了，则直接返回该文件；如果没有找到，则跳转到 /index.html。

```plain
bashCopy code
location /images/ {
    try_files /usr/share/nginx/html/images/logo.png /index.html;
}
```

需要注意的是，如果指定的文件是相对路径，则相对于 root 指令所指定的根目录进行查找；如果文件是绝对路径，则直接查找该文件。另外，如果指定的 URI 是以 @ 符号开头的，则会跳转到一个自定义的 location。



##### 2、$uri
**$uri 是 nginx 变量之一，表示客户端请求的 URI（Uniform Resource Identifier），也就是 HTTP 请求中的路径部分。**

例如，如果客户端请求的是 http://example.com/index.html，则 $uri 的值为 /index.html。

在 nginx 配置文件中，$uri 经常用于指定要匹配的 URI，例如：

```plain
bashCopy code
location /images/ {
    try_files $uri /index.html;
}
```

这个配置表示，如果客户端请求的 URI 包含 /images/，则会尝试返回该 URI 对应的文件。如果该文件不存在，则会返回 /index.html。



##### 3、$router
$router 是一个 nginx 变量，它用于在 nginx 配置文件中的 try_files 指令中，尝试匹配指定的 URI，如果 URI 无法匹配到任何静态资源，则跳转到路由控制器。在上面的 nginx 配置文件中，try_files $uri $uri/ @router; 用于尝试寻找静态文件，如果找不到则跳转到 @router 控制器，即下面的 location @router { ... } 部分。在该部分中，使用 rewrite 指令将请求重定向到 index.html 页面，以启动 Vue 路由控制器。

需要注意的是，$router 变量并不是 nginx 的内置变量，而是 Vue Router 提供的一个变量。在 Vue Router 中，我们通常将 mode 设置为 history 模式时，需要在 nginx 配置文件中配置 $router 变量，以保证路由控制器能够正常工作。



##### 4、return
在 Nginx 中，return 指令用于返回给客户端一个指定的 HTTP 状态码和响应体。当 Nginx 处理一个客户端请求时，如果请求的 URI 符合 return 指令中指定的条件，则 Nginx 将会立即向客户端返回指定的状态码和响应体，而不再继续处理该请求。

例如，在以下配置中，当客户端请求 /denied 时，Nginx 将会返回一个 403 状态码，表示该请求被禁止访问：

```plain
bashCopy code
location /denied {
    return 403;
}
```

可以使用 return 指令返回其他状态码，例如：

+ 301：永久重定向
+ 302：临时重定向
+ 404：页面不存在
+ 500：服务器内部错误等

需要注意的是，如果 return 指令同时指定了状态码和响应体，则 Nginx 会返回状态码和响应体给客户端，而忽略客户端发来的请求体（POST、PUT 等方法的请求体）。



##### 5、location
在Nginx中，location指令是用于匹配请求URI的，语法如下：

```plain
location [标志] 匹配字符串 { ... }
```



几种常见的匹配规则：

=：  精确匹配。优先级最高

^~： 前缀匹配，优先级其次

~ ： 正则匹配，优先级再次 (~* 只是不区分大小写，不单列)

/： 通用前缀匹配，优先级再次



1. 精确匹配

当location指令的匹配字符串是一个等于号（=）后面跟一个字符串时，Nginx会将请求URI与该字符串进行精确匹配，只有当请求URI与该字符串完全相同时，才会使用该location块中的配置。

```plain
location = /hello {
  # 针对 /hello 的请求会使用该块中的配置
}
```



2. 前缀字符串匹配规则

对于 ^~，它表示“URI 前缀匹配”，即如果请求的 URI 与 URI 前缀匹配，就停止搜索正则表达式，直接使用该 location 指令进行处理。此外，^~ 也可以用于提高配置的性能，因为它能够避免在匹配其他 location 指令时进行正则表达式的计算，从而减轻服务器的负载。

```plain
location ^~ /images/ {
    # 处理图片请求
}
```



3. 正则表达式匹配规则

当location指令的匹配字符串是一个正则表达式时，Nginx会将请求URI与该正则表达式进行匹配，只有当请求URI符合该正则表达式时，才会使用该location块中的配置。

~：使用正则表达式进行匹配，区分大小写

+ ~*：使用正则表达式进行匹配，不区分大小写。

```plain
location ~ \.(jpg|png|gif)$ {
  # 针对以 .jpg、.png 或 .gif 结尾的请求会使用该块中的配置
}

```



4. 通用前缀

当location指令的匹配字符串以斜杠（/）结尾时，Nginx会将请求URI与该字符串进行前缀匹配，只有当请求URI以该字符串开头时，才会使用该location块中的配置。

```plain
location /hello {
  # 针对 /hello/ 及其子路径的请求会使用该块中的配置
}

```
