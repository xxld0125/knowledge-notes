使用痛点就不说了，用过的人自然知道，无非就是那几个方面的痛点问题

这里主要讲讲目前客服这些PaaS化的应用，仍旧存在的一些局限问题或者说缺点不足



+ UI样式问题
    - 平台组件开放的样式调整能力太有限，有些UI效果无法实现
+ 交互问题
    - 组件的事件类型和钩子太少，有些交互也无法实现，比如事件前的钩子的处理场景
+ 个性化组件太多
    - 依赖的平台组件过少，过多的依赖了自定义开发的个性化组件，导致维护成本很高
    - 不过 这个问题至少只是成本增高，不至于像其他的玩不下去，无法实现
+ 线上问题难以排查定位
    - 因为不是像传统的本地代码开发方式，无法直接打断点，打日志来定位，排查基本靠猜
+ 扩展应用粒度、页面个性化粒度太高
    - 针对一些微小的个性化需求，只能走扩展应用模式，页面粒度的个性化，导致维护、同步的成本也很高
+ SaaS环境很难玩个性化需求
    - SaaS环境后端就一套代码，如果用租户扩展应用模式来实现个性化需求，一个扩展应用一个新的站点地址，现有的后端逻辑和离线App项目无法支持，需要改造针对租户级别配置PaaS应用站点地址
+ 客服工单系统在越秀环境目前是离线包走App热更新模式
    - 注意：越秀使用客服工单系统，是离线+在线两种模式均有使用，因此目前每次发布，除了下载离线包发热更新外，PaaS平台也需要发布
    - 离线包其实是下载构建后的资源包，这意味着不同分支的合并操作仍旧会有冲突问题
    - 而且切个访问的后端接口环境，都需要重新打包，操作繁琐平台的分支开发模式功能不够完善
+ 平台的分支开发模式功能不够完善
    - 平台只支持建分支，不支持合并时的冲突解决，导致合并分支时，如果出现冲突，解决成本太大，要么学习元数据意义，要么直接废弃某个分支，把废弃分支上的修改全部手动在新分支上再操作一遍
