#### 一、服务端热更新流程
##### 1.热更新表设计
```sql
CREATE TABLE `app_web_version` (
  `id` char(36) NOT NULL COMMENT 'id',
  `app_code` varchar(50) NOT NULL DEFAULT '' COMMENT '应用code',
  `web_version` varchar(20) NOT NULL COMMENT 'web版本号（小版本）',
  `app_version_code_ios` varchar(20) NOT NULL COMMENT 'app版本_iOS',
  `app_version_code_android` varchar(20) NOT NULL COMMENT 'app版本_android',
  `download_url` varchar(500) NOT NULL COMMENT '更新包下载url',
  `update_info` varchar(500) DEFAULT NULL COMMENT '更新说明',
  `release_date` datetime NOT NULL COMMENT '发布日期',
  `created_on` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录创建时间',
  `created_by` char(36) DEFAULT NULL COMMENT '记录创建者Id',
  `modified_on` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '记录修改时间',
  `modified_by` char(36) DEFAULT NULL COMMENT '记录修改者Id',
  `is_deleted` tinyint(1) DEFAULT '0' COMMENT '是否已删除',
  PRIMARY KEY (`id`),
  KEY `idx_app_code` (`app_code`),
  KEY `idx_app_code_app_version` (`app_code`, `app_version`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8 COMMENT='APP web 版本表';
```



##### 2.添加版本记录
基于上面的版本表，我们发布一个版本，就是添加一条记录，为了保证每个大版本只有一个活跃的小版本，还需要删除之前该大版本下的小版本，而且还要让小版本号自增，这块我们可以用 sql 来实现：

```sql
-- 1、 获取新的热更新版本号 --
select @curVersionCode := ifnull(max(version_code), '00')
from app_web_version
where app_code = 'AppCode'
and is_deleted = 0
and (app_version_code_ios = 'AppVersion' or app_version_code_android = 'AppVersion')
;

select @newVersionCode := case when @curVersionCode + 1 > 9 then @curVersionCode + 1 else lpad(@curVersionCode + 1, 2, '0') end
;


-- 2、 作废当前大版本之前的热更新版本 --
update m_app_web_version
set is_deleted = 1, modified_on = now(), modified_by = 'system'
where app_code = 'AppCode'
and is_deleted = 0
and (app_version_code_ios = 'AppVersion' or app_version_code_android = 'AppVersion')
;


-- 3、 生成当前版本的新热更新版本 --
insert into m_app_web_version (id, app_code, web_version, app_version_code_ios, app_version_code_android, download_url, update_info, update_info_url, release_date, created_on, created_by, modified_on, modified_by, is_deleted)
values (uuid(), 'AppCode', @newVersionCode, 'AppVersion', 'AppVersion', 'url', 'updateInfo', 'updateInfoUrl', 'releaseDate', now(), 'system', now(), 'system', 0)
;
```



##### 3.数据下发
在整个热更新的过程中，服务端只负责根据 APP 端提交的参数获取对应的数据版本，并处理更新说明中的换行符以便前端可以简单的做到显示处理。

服务端只要提供一个`get-web-version`的接口即可，接口要求传入三个参数：AppCode、version、platform，这里需要根据platform生成版本的查询字段app_version_code_${platform}，查询的 sql 如下：

```sql
select id, web_version, download_url, update_info,update_info_url from app_web_version where app_code='AppCode' and pp_version_code_{platform} = 'version' and is_deleted = 0;
```

#### 三、灰度热更新
##### 1.灰度热更新的目的
上面介绍的标准热更新流程中，热更新表中的数据只有一条最新的记录是生效的，其他都为失效状态。这样是为了保持所有用户使用的版本都是一样的，且最新。

但是部分场景下我们都需要灰度热更新，比如新功能的发布，一般是需要进行灰度发布，控制出现问题后的影响范围，这个时候标准热更新就无法满足此条件。



##### 2.支持灰度的热更新表设计
相对于标准的热更新表，增加了匹配规则的字段 `rule` ：这个规则要求是一段 JSON 字符串，比如想要为指定用户发布热更新只需要传入规则。

例如下面的灰度规则，指定了 aaa 租户下的 bbb 用户，这样服务端就可以根据请求的数据判断是否命中规则，命中则下发热更新版，没命中则不下发。

```json
{
  "tenantCode":	["aaa"],
  "account": ["bbb"]
}
```



如果只增加一个rule字段，还存在一个问题。因为每个热更新版本只会有一个活跃的小版本，那么发布了灰度版本之后，线上就只有一个灰度版本，如果这时候有一个代码不是上一个热更新版本的用户或者是新安装的用户打开 APP，那么该用户就无法更新到最新的代码，会导致用户无法使用或者是大量的数据问题等等。

这个模式虽然解决了灰度匹配，但是又引出新的问题，我们得升级这个灰度模式。要解决上面的问题，我们需要让一个大版本同时存在多个活跃的小版本，但是这么一来，服务端就无法知道要下发哪一个版本，所以还需要对小版本做优先级划分，根据优先级来返回命中的版本。

所以需要多加一个优先级字段 sort，获取的时候会根据sort做升序排序，然后服务端一次匹配灰度规则，如果灰度规则为空则表示是一个通用的版本。简单描述下这种灰度模式的流程：

```sql
CREATE TABLE `app_web_version` (
  `id` char(36) NOT NULL COMMENT 'id',
  `rule` text DEFAULT NULL COMMENT '灰度规则',
  `app_code` varchar(50) NOT NULL DEFAULT '' COMMENT '应用code',
  `web_version` varchar(20) NOT NULL COMMENT 'web版本号（小版本）',
  `app_version_code_ios` varchar(20) NOT NULL COMMENT 'app版本_iOS',
  `app_version_code_android` varchar(20) NOT NULL COMMENT 'app版本_android',
  `download_url` varchar(500) NOT NULL COMMENT '更新包下载url',
  `update_info` varchar(500) DEFAULT NULL COMMENT '更新说明',
  `sort` int(11) DEFAULT NULL COMMENT '规则判断顺序',
  `release_date` datetime NOT NULL COMMENT '发布日期',
  `created_on` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录创建时间',
  `created_by` char(36) DEFAULT NULL COMMENT '记录创建者Id',
  `modified_on` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '记录修改时间',
  `modified_by` char(36) DEFAULT NULL COMMENT '记录修改者Id',
  `is_deleted` tinyint(1) DEFAULT '0' COMMENT '是否已删除',
  PRIMARY KEY (`id`),
  KEY `idx_app_code` (`app_code`),
  KEY `idx_app_code_app_version` (`app_code`, `app_version`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8 COMMENT='APP web 版本表';
```
