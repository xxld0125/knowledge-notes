<font style="color:rgb(23, 43, 77);">kfpack 指的是在老项目（customer-service，vengineer，v-service-online）里写的 vue 单文件代码的构建工具</font>

# <font style="color:rgb(23, 43, 77);">为什么会有 kfpack</font>
<font style="color:rgb(23, 43, 77);">这个构建工具不是我搞出来的，在我一年前来的时候，它已经存在了，但我更新过这个构建工具一版，再加上一年的经验，我基本能理解为什么会搞出这么一个 kfpack 的工具来</font>

+ <font style="color:rgb(23, 43, 77);">最开始的起因应该是经手的人不想用 jQuery 来写</font>
+ <font style="color:rgb(23, 43, 77);">所以他尝试直接引入 vue 框架的在线编译版的方案</font>
    - <font style="color:rgb(23, 43, 77);">你应该可以找到一些直接在 php 文件里写 vue 语法的代码</font>
+ <font style="color:rgb(23, 43, 77);">但这个方案有个缺点，浏览器兼容问题，因为老项目没有工程概念，写什么代码，浏览器原原本本运行什么代码</font>
+ <font style="color:rgb(23, 43, 77);">而且这个时期，我们的租户应该还有很多使用 IE 浏览器，这就导致，ES6 语法也不能使用</font>
+ <font style="color:rgb(23, 43, 77);">于是乎，kfpack 工具就诞生了</font>
    - <font style="color:rgb(23, 43, 77);">通过与后端配合和约定使用规范，我们可以在代码里写 vue 语法，写 ES6 语法，借助 kfpack 工具构建指定的这一部分代码，构建后的代码以约定的规范输出，php 里以约定的方式来引入使用</font>
+ <font style="color:rgb(23, 43, 77);">但 kfpack 工具应该诞生在很早期，因为我看他不支持 vue 单文件语法，UI 样式上使用的也是 BootStrap 那套</font>
    - <font style="color:rgb(23, 43, 77);">你应该可以找到一些虽然是 vue 语法，但却不是 vue 单文件形式，而是 js, html, css 形式的代码</font>
+ <font style="color:rgb(23, 43, 77);">于是我扩展了他，支持了 vue 单文件语法，以及 element-ui 组件库</font>
    - <font style="color:rgb(23, 43, 77);">这之后，就可以直接用 vue 单文件语法写代码了</font>

# <font style="color:rgb(23, 43, 77);">使用痛点（缺点和不足）</font>
+ <font style="color:rgb(23, 43, 77);">每次写完 vue 代码，必须手动执行 kfpack 命令构建生成代码后，才能看到调试效果</font>
    - <font style="color:rgb(23, 43, 77);">其实我个人觉得这点还好，因为每次构建都是指定一部分代码，基本在秒级内都能构建完毕，比较麻烦的是构建命令需要接收入口文件路径，这个可能不熟练的人找起来就觉得有些麻烦</font>
    - <font style="color:rgb(23, 43, 77);">而且，因为我个人不觉得这点是难以接受的，所以在使用文档里，我都只介绍了这种手动构建的方案，但其实这个构建工具有另一个 kfpack watch 命令，有时间建议你可以抽空研究看看，也许可以解决你这个痛点</font>
+ <font style="color:rgb(23, 43, 77);">提交代码需要连同构建后的代码也提交，这将带来一系列协作问题</font>
    - <font style="color:rgb(23, 43, 77);">由于需要提交构建后的代码，这就导致，如果多分支上改动了同个功能模块，提交了同模块的构建后代码，或者多人协作时，代码必定会冲突</font>
    - <font style="color:rgb(23, 43, 77);">而冲突解决方式很难搞，因为两边的构建后代码都只包含各自的部分，而且因为构建后代码基本都压缩、混淆成一行代码，git 完全无法判断怎么合并，所以才会必定冲突</font>
    - <font style="color:rgb(23, 43, 77);">冲突解决方式就是，冲突的文件随便接受一方，先把源码合并好，然后再重新用 kfpack 工具构建一遍生成新的代码，但这其实也意味着，下次合并必定会再冲突，因为重新构建的这次提交让这个分支的代码走得比原分支快了</font>

# <font style="color:rgb(23, 43, 77);">优点和好处</font>
<font style="color:rgb(23, 43, 77);">这个工具的好处就是可以在老项目里用新语法，用 vue 语法来开发，而不用再依赖于 jQuery 这套老技术栈</font>

<font style="color:rgb(23, 43, 77);">我觉得单就这一点好处，足够盖过他所带来的缺点了</font>

<font style="color:rgb(23, 43, 77);">可能你会疑惑，直接用 vue2 全家桶的技术栈，在新项目里重构不是更好吗？</font>

<font style="color:rgb(23, 43, 77);">没错，这是最理想、最佳的实践方案</font>

<font style="color:rgb(23, 43, 77);">但往往，这是可遇不可求的，因为一次重构，需要后端配合你迁移接口，需要测试投入来资源来全量验证你重构的模块</font>

<font style="color:rgb(23, 43, 77);">而且，这也需要你对当前页面的业务足够了解，因为这种方案最小的重构粒度是页面级，你要有把握对整个页面的所有功能都能重构过去，不留遗漏，否则就又会带来新的债务问题了（重构一半）</font>

<font style="color:rgb(23, 43, 77);">而且而且，在业务组，需求这么赶的情况下，PO基本不会给你时间来排期重构的</font>

<font style="color:rgb(23, 43, 77);">所以，用 kfpack，你可以组件级别的重构，一次次都慢慢的重构当前页面一小部分，等将来有时间，整个页面重构时，直接把这些代码拷到新项目里复用即可</font>

# <font style="color:rgb(23, 43, 77);">后面怎么玩</font>
<font style="color:rgb(23, 43, 77);">这个 kfpack 肯定还会持续很长一段时间的，直到所有老项目里的代码都被重构到新项目去之前，他都会一直存在的</font>

<font style="color:rgb(23, 43, 77);">所以，对于怎么继续更好，更没有痛点的把 kfpack 玩下去，针对上面的两个使用痛点，我思考了下，有些方案建议，你们可以试试：</font>

+ <font style="color:rgb(23, 43, 77);">针对本地开发时，每次写完代码都需要手动构建再能看到调试效果这个问题</font>
    - <font style="color:rgb(23, 43, 77);">要么忍一忍，要么研究看看 kfpack watch 命令，再要么直接研究下脚本去扩展支持下 watch 机制</font>
+ <font style="color:rgb(23, 43, 77);">针对提交了构建后的代码导致合并操作时冲突的问题</font>
    - <font style="color:rgb(23, 43, 77);">这个可以参考 vengineer 项目，把 kfpack 的构建操作加入构建脚本里，让 Dev-Ops 构建平台来执行</font>
        * <font style="color:rgb(23, 43, 77);">这样提交代码时，可以只提交源码，不必提交构建后的代码</font>
        * <font style="color:rgb(23, 43, 77);">这个方案，我尝试去改造过，但遇到一些问题，没有调试通过，后面因为没有时间跟进，就中断了，可以简单看看 hotfix-npm 这个分支，继续去研究下看看</font>
        * <font style="color:rgb(23, 43, 77);">有一些注意点，通过构建平台来构建的话，就只能进行全量构建了，因为平台没法人为主观的只构建指定入口的部分文件</font>
        * <font style="color:rgb(23, 43, 77);">全量构建的话，yl_resource.json 里配置引入的一些构建后的代码文件，已经找不到对应的源码在哪里了，所以需要确认验证下这部分功能是否会受影响，是否需要特殊处理</font>
        * <font style="color:rgb(23, 43, 77);">由于 kfpack 构建脚本好像没有处理好构建失败的场景，导致即时构建失败，构建平台好像也没有中断，会继续下去，这样很有可能会导致发布上生产时，才发现功能异常，可能排查半天才发现是构建平台构建失败了，所以需要想想怎么来检测这种构建失败的场景</font>
        * <font style="color:rgb(23, 43, 77);">hotfix-npm 分支里，在构建平台构建运行的 kfpack 工具，我是用的下载到当前项目级别的方案，但运行时各种报错问题，没有去深究不清楚是不是这种方案导致的，可以试试直接使用构建服务器上的全局 kfpack 工具，但全局也会有一种问题，就是后续你更新 kfpack 的话，需要通知运维去构建服务器上更新下这个全局工具，因为我们没有权限操作</font>
