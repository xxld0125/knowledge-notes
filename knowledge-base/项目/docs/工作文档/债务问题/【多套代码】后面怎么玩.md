# <font style="color:rgb(23, 43, 77);">为什么会有多套代码存在</font>
+ <font style="color:rgb(23, 43, 77);">最早期，pc 端就 customer-service 和 kffrontend，这个不重叠。app 端就 appEngineerKit 和 vengineer，这个其实也可以说是不重叠。业主端就一个 v-service-online。</font>
+ <font style="color:rgb(23, 43, 77);">后来，我为了让App端和业主端的新页面，可以用 vue 技术栈来实现，于是我在 vengineer 和 v-service-online 两个项目里，都加了 web-src 目录，里面创建了用 vue 技术栈初始化的项目。以复用原站点的方案来让新页面有可以用 vue 技术栈写的渠道。</font>
    - <font style="color:rgb(23, 43, 77);">复用原站点方式，其实有多个好处，一个是因为仍旧跟后端同站点，所以他们接口的校验仍旧可以使用 cookie 方案，不需要后端去做接口迁移，支持 token 的机制。这个好处就是，我们甚至可以直接调用老接口来重构，不需要依赖后端迁移接口。</font>
    - <font style="color:rgb(23, 43, 77);">再有就是，不需要部署新站点，占用新资源，新域名，这样也就不需要向上升级去搞新增站点应用的模式</font>
    - <font style="color:rgb(23, 43, 77);">这时候新加入的这两个项目仓库，本质上其实也跟原先的没有重叠，只是新增了一种渠道来提供新页面可以写新代码。</font>
+ <font style="color:rgb(23, 43, 77);">再后来，因为 appEngineerKit 这个项目三个业务组团队高耦合在一起，发布经常出现问题，为了解耦，公共组搞出了 app-customer-service </font>
    - <font style="color:rgb(23, 43, 77);">app-customer-service 相当于把 appEngineerKit 的代码拷贝到新的仓库，用在线方式部署上去</font>
    - <font style="color:rgb(23, 43, 77);">新的明源智慧客服 APP 就以 H5 的方式集成进去，这样客服代码就没有耦合在 App 端的项目里</font>
+ <font style="color:rgb(23, 43, 77);">再后来，上面推广 paas，战略团队直接用 paas 重构了整个工单模块（日常维保、任务登记、任务详情）</font>

<font style="color:rgb(23, 43, 77);">由于后面搞出来的这些项目，推广都只推广了一半，做也是差不多做一半，导致没法直接替换掉原版，就造成目前多套代码同时维护的问题存在了</font>
